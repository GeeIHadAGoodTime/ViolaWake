# S1.3 Requirements Synthesis: Competitive Audit -> Product Requirements

## 1. Product Vision
Wakeword is a Python-first, on-device voice SDK for developers who need production wake word detection without proprietary lock-in. The product packages wake word detection, TTS, VAD, and STT into a single `pip install viola-sdk` workflow so teams can add local voice interfaces in a few lines of code. The core differentiator is an open training and evaluation pipeline paired with production ONNX deployment, which gives developers a credible alternative to Picovoice's black-box commercial stack and openWakeWord's less production-hardened offering.

## 2. Competitive Positioning
| Area | Wakeword | Picovoice Porcupine | openWakeWord | Requirement Implication |
| --- | --- | --- | --- | --- |
| Wake word customization | Full training pipeline, ONNX export, Cohen's d / FAR / FRR evaluation, custom domain adaptation | Custom wake words, but proprietary and metered | Open-source inference, but weaker training/productization | Product must expose custom wake word workflows as a first-class differentiator. |
| Measured quality | Current internal baseline: Cohen's d 15.10 on synthetic negatives; speech-negative d-prime TBD | Commercial-grade, but opaque quality reporting | Community traction, less published production hardening | Product must publish repeatable benchmarks and operating points. |
| Packaging model | First-class Python SDK | Python SDK wraps a C binary | Python/open-source, but fragmented setup | Product must ship clean Python APIs and PyPI packaging. |
| Business model | Open core, local inference, commercial upsell | Proprietary, metered, K+/yr commercial licensing | Free/open-source | Product must undercut the pricing and adoption friction of Picovoice while keeping paid expansion paths. |
| Voice stack breadth | Wake + TTS + VAD + STT in one SDK | Wake word, VAD, TTS, STT, NLU, LLM | Mostly wake word focused | Product should win the niche SDK position by shipping the practical local voice stack, not a broader assistant platform. |
| TTS position | Apache-2.0 Kokoro-82M, streaming, production-tested | Proprietary Orca | No equivalent integrated TTS story | Product should bundle TTS as a built-in reason to adopt the SDK. |
| Platform position | Realistic Python-first rollout, WASM next | 12+ platforms today | Primarily developer/community use | Product must prioritize a credible platform sequence instead of overpromising parity at launch. |

## 3. Feature Requirements by Priority
| Feature | Priority | Effort | Concrete Requirement | Competitive Justification |
| --- | --- | --- | --- | --- |
| Wake word runtime SDK | MUST | High | Extract ViolaWake into a standalone Python package with a stable API for model load, streaming inference, sensitivity control, callbacks/events, and ONNX runtime execution on Windows and Linux. | This is the strongest product differentiator and directly answers Picovoice's proprietary model and C-backed Python wrapper. |
| Wake word training/export/eval pipeline | MUST | High | Ship training, ONNX export, and Cohen's d/FAR/FRR evaluation workflows as supported developer tooling, not internal-only scripts. | The open training pipeline is the clearest advantage over Picovoice and the production-hardening gap versus openWakeWord. |
| PyPI packaging and install path | MUST | Medium | Publish `viola-sdk` to PyPI with pinned extras for `wake`, `tts`, `vad`, and `stt`; installation on Windows and Linux must succeed from clean environments. | The recommended strategy is a 5-line Python-first SDK; that fails without frictionless packaging. |
| SDK documentation | MUST | Medium | Publish quickstart, API reference, benchmark page, model versioning policy, and at least three end-to-end example apps. | Documentation is called out as an infrastructure gap and is required to convert competitive differentiation into adoption. |
| Accuracy benchmark publication | MUST | Small | Publish the reference model's Cohen's d score, FAR/FRR curves, test conditions, and default operating point in repo docs and release notes, while clearly separating synthetic-negative results from speech-negative benchmarking. | Competitors rely on black-box claims; transparent measurement makes the open model credible. |
| Model versioning | MUST | Small | Version runtime APIs and shipped models independently using SemVer-compatible release tags, with upgrade notes for model behavior changes. | Once benchmarks are public and the SDK is packaged, model regressions become a product risk that must be managed explicitly. |
| Kokoro TTS SDK | MUST | Medium | Package Kokoro-82M as a supported SDK module with sentence-chunked streaming, voice selection, file output, and in-memory streaming APIs. | Apache-2.0 TTS is a direct advantage over Picovoice's proprietary Orca and strengthens the "voice stack in one SDK" story. |
| VAD module | SHOULD | Medium | Extract the existing VAD path into a standalone module with frame-level speech events, threshold controls, and wake-word-pipeline integration hooks. | VAD is part of the niche SDK bundle that makes the product more useful than a wake-word-only library. |
| Whisper STT wrapper | SHOULD | Medium | Package the faster-whisper wrapper with batch transcription and streaming transcription APIs that share the SDK's audio interfaces. | STT increases product completeness and helps position the SDK against Picovoice's broader catalog without chasing NLU/LLM scope. |
| Commercial license infrastructure | SHOULD | Medium | Implement license key checks, usage terms, and release packaging for a paid commercial tier without degrading local inference in the free tier. | Open core only works if the commercial path is real, and pricing is the weakness we are explicitly exploiting. |
| DeepFilterNet noise suppression integration | COULD | Medium | Add ONNX-based preprocessing as an optional pipeline module that can be enabled ahead of wake word and STT stages. | This improves domain robustness using existing Apache-2.0 tooling without spending effort on new model research. |
| Custom wake word console | COULD | Large | Build a web UI for dataset upload, training job creation, benchmark review, and model export once the Python SDK foundation is stable. | The console has high developer value, but it is a multiplier on the training pipeline, not a prerequisite for SDK launch. |
| JavaScript/Node SDK | COULD | Large | Delay native JS packaging until the WASM runtime path is proven. | The competitive audit recommends a realistic platform path; overextending here would slow the core differentiators. |
| Speaker recognition / diarization | WON'T (v1) | Large | Do not allocate v1 engineering time to diarization or speaker identity features. | This is a different market from wake word activation and does not improve the Picovoice replacement story. |
| Audio NLU / picoLLM-equivalent | WON'T (v1) | Large | Do not build a Rhino-equivalent intent engine or an LLM runtime. Integrate with Ollama or llama.cpp instead. | Open-source options already outperform picoLLM-class tooling, and this scope does not strengthen the main wedge. |

## 4. MVP Scope (v1.0)
The v1.0 release target is a 3-4 month Python SDK MVP that ships a practical local voice stack, not a multi-platform assistant platform.

What ships in v1.0:
- `viola-sdk` on PyPI for Windows and Linux.
- A wake word module backed by ViolaWake ONNX inference with streaming microphone/file input, configurable sensitivity, event callbacks, and bundled reference models.
- Supported developer tooling for custom wake word training, ONNX export, and benchmark generation using Cohen's d plus FAR/FRR outputs.
- A Kokoro TTS module with sentence-chunked streaming, voice selection, and file/stream output APIs.
- A VAD module with speech start/stop events and direct compatibility with the wake and STT modules.
- A Whisper STT module with batch transcription plus streaming transcription mode.
- SDK docs, quickstarts, API reference, examples, benchmark documentation, model versioning policy, and release notes.
- Commercial-tier scaffolding: license enforcement, packaging split, and pricing-ready release mechanics.

What does not ship in v1.0:
- JavaScript/Node SDK.
- Mobile SDKs.
- Raspberry Pi and ARM optimization work beyond basic smoke testing.
- Custom wake word console.
- Speaker recognition, diarization, audio NLU, or LLM runtime features.

## 5. Platform Roadmap
| Target Window | Delivery Target |
| --- | --- |
| Month 1 | Extract ViolaWake runtime from Viola, define Python API surface, stand up package skeleton, and lock benchmark methodology. |
| Month 2 | Ship alpha wake word + Kokoro TTS modules on Windows/Linux with docs, examples, and internal benchmark automation. |
| Month 3 | Add VAD and streaming STT, finalize model versioning and commercial license infrastructure, and publish benchmark results. |
| Month 4 | Release v1.0 GA on PyPI for Windows/Linux with wake + TTS + VAD + STT, examples, and production packaging. |
| Months 5-8 | Port ViolaWake and Kokoro to WASM via ONNX Runtime Web, then expose a JavaScript/Node SDK that wraps the WASM path. |
| Months 9-12 | Add Docker distribution, Raspberry Pi and ARM optimization, and deployment guidance for edge devices. |
| Post-12 months | Evaluate mobile delivery via React Native + WASM or native JNI/Swift only after Python, WASM, and ARM usage validates the demand. |

## 6. Open Core Business Model
| Free / Open Core | Paid / Commercial |
| --- | --- |
| Local Python SDK runtime for wake word inference with reference/community models | Commercial license for proprietary deployment and closed-distribution use |
| Kokoro TTS module, VAD module, and Whisper STT wrapper for local development | Custom wake word training workflows packaged for business use and supported deployment |
| Basic docs, quickstarts, examples, benchmark methodology, and model version history | Hosted or managed training workflows, premium support, SLAs, and enterprise packaging |
| Transparent benchmark publication and model evaluation outputs | Custom wake word console, team features, and deployment controls |

Business model requirements:
- Free tier adoption must be frictionless enough to counter Picovoice's K+/yr pricing barrier.
- Paid tier value must come from commercialization, workflow acceleration, and support, not from crippling local inference in the free product.
- Pricing and packaging must preserve a clear "better than proprietary metering" message.

## 7. Success Metrics
| Metric | v1.0 Requirement | Why It Matters |
| --- | --- | --- |
| Wake word quality | Maintain or exceed the current Cohen's d 15.10 synthetic-negative baseline while adding a real speech-negative benchmark. | The wake word engine is the primary product wedge; regression here breaks the strategy. |
| Transparency | Publish benchmark inputs, operating point, and FAR/FRR curves for every reference model release. | This is how the SDK differentiates from black-box competitors. |
| False accepts | Default operating point target: FAR <= 1 false accept per 24 hours per always-on device in the reference benchmark. | Competitive wake word products are judged on trust; false wakes destroy perceived quality. |
| False rejects | Default operating point target: FRR <= 5% on the reference test corpus at the published default sensitivity. | Developers need a usable default that does not require immediate retuning. |
| Wake inference latency | <= 50 ms processing time per 1 second audio window on a standard x86 laptop CPU reference target. | Local UX depends on low-latency detection without specialized hardware. |
| TTS startup latency | First audio chunk <= 300 ms for a short sentence on the same reference target. | TTS only strengthens the bundle if it feels responsive. |
| STT streaming latency | First partial transcript <= 500 ms after speech onset on the same reference target. | Streaming STT needs to feel interactive to justify bundling it. |
| Installation success | Clean install and example app success on supported Windows and Linux environments before every release. | The Python-first wedge fails if packaging is brittle. |
| Documentation usability | A new developer can reach first wake event from the quickstart in <= 10 minutes. | Ease of adoption is a direct response to proprietary friction. |

## 8. What We Skip and Why
| Skipped for v1 | Reason |
| --- | --- |
| Rhino-equivalent audio NLU | Research-heavy scope that does not improve the core wake word SDK wedge and is explicitly identified as out of market for this phase. |
| picoLLM-equivalent runtime | Ollama and llama.cpp already cover this space better for free; integration is cheaper than competition. |
| Speaker recognition / diarization | Different buyer and workflow from wake word activation; low leverage for the first release. |
| Broad platform parity with Picovoice at launch | The realistic path is Python-first, then WASM, then Node, then ARM/mobile; pretending otherwise increases delivery risk. |
| Mobile-native SDKs | Too early before Python and WASM adoption validates the need. |
| Full browser and Node launch in v1 | JavaScript should follow proven WASM runtime support, not block the 3-4 month MVP. |
| Custom wake word console in the initial release | Valuable, but secondary to making the underlying training/export/eval pipeline and SDK stable first. |
